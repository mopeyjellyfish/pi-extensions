# Pi Development Workflow

Use this skill for non-trivial features and fixes that need durable decisions and evidence. Read [references/philosophy.md](references/philosophy.md) when shaping a pitch, planning slices, or reviewing design quality; it defines the sourced agent-native Shape Up adaptation behind these rules. When pi-subagents is available, read [references/model-routing.md](references/model-routing.md) before delegating so every child has an explicit cost-appropriate model, effort, context boundary, and escalation rule.

## Route from idea to delivery

1. Read `development_workflow` status first. If none exists, resolve the workspace before starting the ledger: when Worktrunk is available, inspect its status, activate an existing branch or PR, or ask before creating a named feature branch. Otherwise confirm the effective workspace. Only then ask the user to run `/dev-workflow start <title>` or explicitly adopt existing work.
2. **Research before interviewing:** in that routed workspace, read repository instructions, current behavior, relevant code paths, contracts, tests, history, and prior decisions; reproduce or probe when useful. Record fresh `research` evidence for every workflow. Investigate external facts only when they can change the pitch. Summarize implications in the pitch's Research Basis and use bounded linked `research/` artifacts only when the evidence cannot stay concise.
3. Grill the evidence-backed idea in repeated structured batches: identify the motivating story, desired outcome, demand, constraints, and unresolved product decisions. Never ask the user for facts the repository can answer. After each batch, summarize resolved decisions, changed assumptions, and remaining blockers before asking the next dependent batch.
4. **Set boundaries:** agree why the problem is worth the investment and a qualitative agent-investment appetite covering context surface, change depth, uncertainty allowance, validation burden, assurance level, scope cuts, stop conditions, and fixed floors. Record the mandatory wall-clock backstop separately as a circuit breaker, never as the appetite or an effort estimate.
5. **Rough out the elements:** sketch fixed product decisions, agent discretion, macro solution elements, reused seams, boundaries, flows, and observable acceptance signals without over-specifying implementation.
6. **De-risk the concept:** investigate decisive unknowns; give each rabbit hole evidence, containment, and a tripwire; cut generality; and surface decisions the user must own.
7. **Write the pitch:** create `spec.md` as a rough, solved, bounded human-agent contract. Make every section operational for the building agent. When drafting is delegated, use a fresh Sol high planner and keep product decisions with the main orchestrator; escalate to Sol xhigh only on the hard-planning triggers. Read [references/pitch.md](references/pitch.md), grill it, and obtain explicit user agreement before asking them to approve Pitch.
8. Build an evolving `plan.md` and the first integrated demonstrable `VS-*.md` slice. Read [references/slices.md](references/slices.md), preview the slice map and validation contract, then obtain explicit agreement before Plan approval. When delegated, use a fresh Sol high planner, escalating to Sol xhigh only on the defined hard-planning triggers, and make the slice executable by a Terra worker.
9. After Plan approval, execute the well-scoped slices mainly autonomously, one active slice at a time, with RED → GREEN → REFACTOR and focused verification. Keep the main chat as orchestrator and one fresh worker as the writer: Terra medium by default, Terra high for difficult bounded implementation, and Sol medium only after the Sol planner or Oracle revalidates a sound plan whose implementation still needs frontier judgment. Never use Terra xhigh as a substitute for replanning or Sol capability. Do not add routine checkpoints. Stop only when a stated stop condition fires, product or architecture behavior needs a user decision, the approved pitch or No-Gos would change, a material rabbit hole or fixed floor is threatened, the backstop expires, or an external/destructive action needs authorization. Read [references/build.md](references/build.md).
10. Obtain independent evidence where possible, normally with one fresh Sol high reviewer per completed slice, fix accepted findings through the sole Terra writer, and rerun fresh final checks without inserting a human ceremony. Read [references/review-and-ship.md](references/review-and-ship.md).
11. Request each next transition with `development_workflow`. Discover, Build, and Review advance when their evidence gates pass. Only the researched Pitch and first-slice Plan require direct user approval; shipping remains separately authorized.

Use [templates/spec.md](templates/spec.md), [templates/research.md](templates/research.md), [templates/plan.md](templates/plan.md), and [templates/vertical-slice.md](templates/vertical-slice.md) as starting points, not status stores. Create a separate research file only when its decision-changing evidence cannot stay concise in the pitch's Research Basis.

## Operating rules

- The main chat agent is the sole orchestrator and decision authority. Explicitly select every child's model and effort; never allow a delegated child to inherit the expensive Sol parent accidentally. Children return evidence and escalation requests, while the main chat synthesizes results and makes every workflow transition.
- One worker writes by default. Parallelize read-only investigation and genuinely independent review only.
- Use fresh context for scouts, planners, workers, and reviewers with the pitch, active slice, direct contracts, and validation contract. Fork only the triggered Oracle whose purpose is inherited decision-consistency review.
- Use Todo only for work discovered inside the active slice. Never copy pitch, slice, phase, approval, or ledger state into Todo.
- Prefer capability checks: use Worktrunk for isolated worktrees, Question for structured ambiguity, LSP for code intelligence, web search for external research, GitHub for remote evidence/actions, Git conventions for commits, and pi-subagents for explicit-model scouts, planners, workers, reviewers, and triggered Oracle checks when available.
- Route non-trivial work through Worktrunk before starting or adopting the ledger so research evidence is born in the intended workspace. Activate existing work directly; ask before branch creation. Defer hook, routing, and removal mechanics to `pi-worktrunk`.
- With Question, ask 2–4 independent related questions in one batch when possible; do not batch questions with unresolved decision dependencies. If the user redirects to chat, address the clarification and reopen the revised questions with its continuation ID. Without Question, ask the same compact numbered batch conversationally or record an unresolved decision and remain blocked.
- Without Worktrunk, confirm the effective workspace and never create a hidden raw-worktree manager. Without Todo, follow the active slice sequentially and report current/next work without creating another state store.
- Without LSP, use bounded repository search plus compiler, typechecker, and test evidence, and disclose reduced semantic precision. Without web search, use repository-local evidence and state that external research was not performed; never imply current external facts were verified.
- Without Git metadata, label freshness identity unavailable. Without Git conventions, follow repository instructions and preview explicitly authorized mutations. Without GitHub tooling, give the relevant command or URL and wait for separate authorization instead of inventing a client. Without pi-subagents, the main chat plans and implements sequentially, reviews in the same context, and labels that assurance reduced.
- If any optional extension is absent, continue only through its truthful fallback; never recreate its tool.
- Never claim a commit, push, PR, merge, release, or deployment before it occurs. Remote and destructive actions require explicit user authorization.
- Appetite is a human commitment explaining why the problem deserves investment and bounding the qualitative agent envelope: context surface, change depth, uncertainty allowance, validation burden, assurance level, scope control, and fixed floors. This deliberately adapts Shape Up's fixed-time appetite for agent execution. It is not a token budget or universal complexity score. Before Pitch approval, the ledger separately requires an `Nh`, `Nd`, or `Nw` wall-clock backstop only for stale-work circuit breaking. Pausing suspends workflow mutation, not that timer.
- If a stop condition is reached or the shaped investment is exceeded before the timer, stop and ask the user to cut scope, rewind the pitch, or abandon. After a cut, request the evidence-gated Build transition; do not add another approval ceremony. Circuit commands are only for wall-clock backstop expiry; never silently expand investment, extend time, or lower fixed floors.
